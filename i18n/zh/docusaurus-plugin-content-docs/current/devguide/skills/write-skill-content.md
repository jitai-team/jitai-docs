# 写出 Agent 能稳定执行的 Skill 内容

Skill 内容的目标不是“写得像说明书”，而是让 Agent 在合适的场景下稳定使用这套能力。好的 Skill 内容应该清楚说明适用场景、处理步骤、输出格式和约束边界。

## 先写清楚 Skill 适用场景 {#write-applicable-scenarios-first}

开头先说明这个 Skill 适合处理什么问题。Agent 在执行任务时，需要判断当前请求是否应该使用这个 Skill，因此场景描述要具体。

可以这样写：

```markdown
## 适用场景

当用户需要分析客户跟进情况、判断客户优先级、生成下一步跟进建议时，使用本 Skill。
```

避免只写“用于客户管理”“用于数据分析”这类泛化描述。描述越具体，Agent 越容易正确选择。

## 用能力范围限定边界 {#define-capability-scope}

能力范围用于告诉 Agent 这个 Skill 能做什么，也告诉它不要越界。

示例：

```markdown
## 能力范围

- 根据客户最近互动、商机阶段和历史成交情况判断跟进优先级。
- 为销售人员生成下一步沟通建议。
- 将客户风险点整理为清单。
- 不直接修改客户资料，不替代人工做最终成交判断。
```

边界越清楚，Agent 越不容易把 Skill 用到错误场景中。

## 用执行步骤沉淀操作流程 {#write-execution-steps}

执行步骤是 Skill 的核心。它把团队经验变成 Agent 可以遵循的流程。

建议使用有顺序的步骤，而不是大段自然语言。

```markdown
## 执行步骤

1. 先确认用户要分析的客户范围。
2. 检查客户最近一次沟通时间、商机阶段、未完成任务和历史成交记录。
3. 按高、中、低三个等级判断跟进优先级。
4. 对每个客户给出下一步建议。
5. 最后输出需要人工重点关注的风险客户。
```

如果某一步需要工具或数据支持，可以只写业务动作，不需要在 Skill 中展开工具配置。工具暴露属于 Agent 的能力配置，详见[按需向大模型暴露工具函数](../ai-agent/agent-tool-management)。

## 用输出格式稳定结果 {#stabilize-output-format}

输出格式能减少 Agent 每次回答风格不一致的问题。尤其是报表、审核意见、分析结论等场景，建议明确输出结构。

示例：

```markdown
## 输出格式

请按以下结构输出：

### 总体结论
用 2-3 句话概括客户跟进状态。

### 高优先级客户
按客户名称、原因、建议动作列出。

### 风险提醒
列出需要人工介入的风险点。
```

如果 Agent 还需要结构化输出，可以结合[输入输出配置](../ai-agent/agent-input-output)完成。

## 用约束条件减少误用 {#add-constraints}

约束条件用于告诉 Agent 哪些行为不应该发生。

常见约束包括：

- 不要编造不存在的数据。
- 数据不足时先说明缺失信息。
- 不直接承诺价格、合同条款或法律结论。
- 不绕过角色权限执行高风险操作。
- 不输出与业务无关的解释。

示例：

```markdown
## 约束条件

- 如果缺少客户互动记录，必须说明无法判断优先级的原因。
- 只能给出跟进建议，不要直接替销售人员做成交承诺。
- 输出中不要包含与客户无关的内部推理过程。
```

## 用示例提升可执行性 {#use-examples}

示例可以帮助 Agent 理解“好结果长什么样”。对于输出要求复杂的 Skill，建议提供一到两个短示例。

示例不需要覆盖所有情况，但要体现关键规则：

- 输入信息是什么。
- 判断过程关注哪些因素。
- 输出结果应该怎样组织。

如果示例较多，建议拆到附件中维护，详见[用附件管理复杂规则、模板和示例](./skill-attachments)。

## Skill 内容不要替代 Agent 主提示词 {#do-not-replace-agent-system-prompts}

Skill 负责可复用能力，Agent 主提示词负责当前 Agent 的角色、目标和整体行为边界。不要把某个 Agent 的身份设定全部写进 Skill，否则其它 Agent 安装时会产生冲突。

如果你正在设计 Agent 的主提示词，可以参考[编写 System Prompts](../ai-agent/create-react-agent#编写-system-prompts)。
